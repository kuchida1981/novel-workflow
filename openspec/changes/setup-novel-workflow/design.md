## コンテキスト

小説執筆は、物語の規模が大きくなるほど設定の不整合やテーマの乖離といった問題が発生しやすくなります。本プロジェクトでは、ソフトウェア開発における「仕様駆動開発」の考え方を小説執筆に適用し、AIエージェントと人間が協調して一貫性のある物語を構築するためのフレームワーク「Novel Workflow (NW)」を導入します。

## ゴール / 非ゴール

**ゴール:**
- 小説作成の各フェーズ（核、素材、構造、分解、初稿）をディレクトリ構造とファイルで管理する。
- 上流工程の決定事項を下流工程の制約条件として機能させるライフサイクルを確立する。
- ユーザーによる「承認（ゲート）」プロセスを明確にし、ステップごとの品質を担保する。
- ワークフロー自体を振り返るためのポストモーテム・テンプレートを提供する。

**非ゴール:**
- 本フェーズにおいて、特定のAIモデル専用のプロンプトエンジニアリングの詳細を定義すること（汎用的なフレームワークを目指す）。
- 文法チェックや校正の自動化ロジックの完全な実装（まずは枠組みの構築に集中する）。

## 設計決定

### 1. ファイルベースの成果物管理
各ステップの成果物を Markdown ファイルとして `novel/artifacts/` 配下で管理します。
- **理由**: Git による履歴管理が容易であり、テキストベースの AI エージェントとの親和性が高いためです。また、ユーザーがエディタで直接編集することも可能です。

### 2. 線形な依存関係モデル
Step 1 (Core) -> Step 2 (Assets) -> Step 3 (Plot) -> Step 4 (Scenes) -> Step 5 (Draft) の順で進めます。
- **理由**: 小説の「核」が決まらなければ適切な「素材（キャラ）」は選べず、素材が決まらなければ具体的な「構造（プロット）」は組めないという、論理的な依存関係を尊重するためです。

### 3. 手動ゲートによるステータス管理
各ステップの完了は、自動検証だけでなくユーザーの明示的な承認を必要とします。
- **理由**: 創作活動において、最終的な価値判断は人間に委ねるべきであり、納得感を持って次のステップへ進むことがプロジェクトの成功に不可欠であるためです。

## リスク / トレードオフ

- **[リスク] 上流工程への頻繁な回帰**: 下流で新しいアイデアが出た際に、上流を修正すると下流の成果物がすべて無効になる可能性があります。
  - **対策**: `workflow.md` において、変更の影響範囲を最小限にするための「バックトラック・ルール」を定義します。
- **[リスク] 制約による創造性の阻害**: 上流の制約が強すぎると、執筆時の自由な発想が制限される可能性があります。
  - **対策**: 制約を「絶対的な壁」ではなく「ガイドライン」として機能させ、必要に応じて上流を柔軟に更新できる運用（リファクタリング）を許容します。
